## Problem Statement

El generador de puzzles llama `forwardSolveState` desde cero en cada una de las ~50 rondas del loop de síntesis de pistas (`synthesizeClues` en `generator.ts`). Para tableros 9×9+ esto produce millones de llamadas a `evaluateClue` por intento, haciendo que 10×10 y 11×11 sean inviables. El juego actualmente tiene 30 niveles (max 9×9). Este plan agrega levels 31-45 con tableros 10×10 y 11×11 mediante una optimización algorítmica.

## Premises (confirmed by user)

1. La API pública `generatePuzzle(seed, level)` no cambia.
2. Los JSONs en `public/puzzles/level-1..30.json` NO se modifican (generación offline, niveles existentes estables).
3. Los puzzles generados siguen siendo forward-solvable (pura deducción lógica, sin adivinanza).
4. TypeScript estricto (`tsc --strict`). Todos los tests existentes deben pasar.
5. La generación es offline: scripts producen JSON, el juego los sirve estáticamente.
6. `solver.ts` NO cambia (solo `generator.ts` y scripts).

## Implementation Plan

### T1 — `SynthesisState` type + helpers (`src/logic/generator.ts`)

**Nuevos tipos internos** (locales a generator.ts, no exportados):

```typescript
type SynthesisState = {
  candidates: Map<string, Set<CellId>>  // charId → celdas ocupiables válidas restantes
  placed: Record<string, CellId>         // charId → celda fijada
  usedRows: Set<number>
  usedCols: Set<number>
  contradiction: boolean                  // true si algún personaje quedó sin candidatos
}
```

**`buildInitialState(board: Board, characters: Character[]): SynthesisState`**

Construye el estado con todos los personajes apuntando a TODAS las celdas ocupiables del tablero. `placed = {}`, `usedRows = new Set()`, `usedCols = new Set()`, `contradiction = false`.

Requiere pre-computar `cellById: Map<CellId, Cell>` una sola vez en `synthesizeClues` (ver T3). También usar `cellById` en la función interna `getCell` de `synthesizeClues` — reemplazar el doble loop O(N²) por `cellById.get(cellId as string)!` (mismo pattern que `forwardSolveState:243` y `solver.ts:126`).

### T2 — `applyClueToState` con worklist AC-3 (`src/logic/generator.ts`)

```typescript
function applyClueToState(
  state: SynthesisState,
  clue: Clue,
  board: Board,
  characters: Character[],
  clues: Clue[],       // pistas acumuladas hasta ahora (sin incluir la nueva)
  cellById: Map<string, Cell>
): SynthesisState
```

**Algoritmo worklist (AC-3 incremental):**

1. Reducir `state.candidates.get(clue.subject)` eliminando celdas donde `evaluateClue(clue, {[subject]: cellId}, board, 'partial') === false`.
2. Si `.size === 0` → `return { ...state, contradiction: true }`.
3. Inicializar `worklist = new Set([clue.subject])`.
4. **Loop worklist hasta vacío:**
   a. Extraer `charId` del worklist.
   b. Si `state.candidates.get(charId).size === 1 && !state.placed[charId]`:
      - Fijar: `state.placed[charId] = cellId`, agregar fila/col a `usedRows`/`usedCols`.
      - Para cada otro personaje no fijado: eliminar celdas que compartan la fila/col fijada; si cambió → agregar al worklist.
   c. Re-evaluar todas las pistas acumuladas (`clues`) donde `prevClue.subject === charId`:
      - Para cada celda en `candidates[charId]`: si `evaluateClue(prevClue, {...placed, [charId]: cellId}, board, 'partial') === false` → eliminar.
      - Si cambió → agregar `charId` al worklist.
   d. **Arco inverso — propagación al shrink** (no solo al placement): si los candidatos de `charId` cambiaron (antes > después en paso 4c, O si charId fue fijado en 4b), agregar al worklist todos los `prevClue.subject` donde `(prevClue.params as any).otherId === charId`. Esto garantiza que relational clues como `direction_of(X, Y)` o `same_room_as(X, Y)` se re-evalúen para X cuando Y tiene menos candidatos, incluso sin estar fijado.
   e. Si `candidates[charId].size === 0` → `return { ...state, contradiction: true }`.
5. Retornar estado actualizado.

**Nota de implementación:** `candidates` usa `Set<CellId>` (O(1) delete). El estado es mutado in-place dentro de la función (copia del estado se hace en el caller vía `{ ...state, candidates: new Map([...state.candidates].map(...)) }`).

### T3 — Refactorizar `synthesizeClues` (`src/logic/generator.ts`)

Reemplazar el loop actual (líneas 407-473):

```typescript
// OLD — 50 llamadas completas a forwardSolveState:
for (let round = 0; round < 50; round++) {
  const { placed, candidates } = forwardSolveState(board, characters, clues)
  // ... pick best clue ...
  clues.push(bestClue)
}

// NEW — estado incremental:
const cellById = new Map<string, Cell>(board.cells.flat().map(c => [c.id as string, c]))
let state = buildInitialState(board, characters)

for (let round = 0; round < 50; round++) {
  if (Object.keys(state.placed).length === characters.length) break
  if (round === 20 && Object.keys(state.placed).length < Math.max(1, characters.length - 3)) break

  const bestClue = pickBestClue(state, characters, board, solution, clues, rng)
  if (!bestClue) break

  // Copia profunda del Map de candidatos antes de aplicar
  const stateCopy: SynthesisState = {
    ...state,
    candidates: new Map([...state.candidates].map(([k, v]) => [k, new Set(v)])),
    placed: { ...state.placed },
    usedRows: new Set(state.usedRows),
    usedCols: new Set(state.usedCols),
  }
  const nextState = applyClueToState(stateCopy, bestClue, board, characters, clues, cellById)
  if (nextState.contradiction) continue  // pista contradictoria → saltarla

  clues.push(bestClue)
  state = nextState
}
```

### T10 — Tests (`src/test/generator.test.ts` o nuevo archivo)

Agregar tests para la nueva lógica incremental:

1. `buildInitialState`: verifica que todos los personajes tienen candidatos = todas celdas ocupiables.
2. `applyClueToState`: aplicar `in_room` clue → reduce candidatos del sujeto.
3. `applyClueToState`: aplicar clue que fija un personaje → `state.placed` se actualiza.
4. `applyClueToState`: contradicción → `state.contradiction = true`.
5. `applyClueToState`: arco inverso — fijar personaje Y actualiza candidatos de X que tiene `same_room_as(X, Y)`.
6. **[ENG-F2] Test de equivalencia**: para un tablero fijo y un set de pistas conocido, verificar que el estado final de `buildInitialState` + `applyClueToState` × N produce `candidates` y `placed` idénticos a los de `forwardSolveState(board, characters, clues)`. Este es el test de correctitud central.
7. `applyClueToState`: arco inverso al shrink — cuando Y NO está fijado pero sus candidatos se reducen (shrink sin placement), X con `direction_of(X, Y)` también reduce candidatos.
8. `generatePuzzle(seed, 31)` genera un puzzle válido (10×10, forward-solvable).
9. `generatePuzzle(seed, 38)` genera un puzzle válido (11×11, forward-solvable).
